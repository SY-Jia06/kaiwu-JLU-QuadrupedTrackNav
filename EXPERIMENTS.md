# 实验记录

这个文件记录 PPO 实验改动和腾讯开悟平台评测结果。后续尽量保持同一套评测口径，方便横向比较，也方便答辩时说明调参路径。

## 当前评测口径

- 开始记录日期：2026-05-08
- 算法路径：`agent_ppo`
- 评测模式：`standard`
- 评测难度档：`[0, 1, 3, 4]`
- 评测地形：`pyramid_slope`、`pyramid_slope_inv`、`pyramid_stairs`、`pyramid_stairs_inv`
- 备注：上表主要记录当前截图中的 4 地形 standard 口径；初赛/天梯 standard 口径还可能包含 `maze`。后续如果使用含 maze 的评测，需要单独标注为 5 地形口径，不能和 4 地形结果直接混在一起比较。
- 评测命令：

```toml
[commands.limit]
lin_vel_x = [0.0, 1.0]
lin_vel_y = [-0.0, 0.0]
ang_vel_z = [0.0, 0.0]
```

### 决赛 Track 口径

2026-05-27 起，决赛口径切换为 track mode，难度档覆盖 `[0, 1, 3, 4, 5, 6, 7, 8, 9]`，命令上限为 `lin_vel_x=[0.0,0.8]`、`lin_vel_y=[-0.3,0.3]`、`ang_vel_z=[-1.5,1.5]`。该口径会显著暴露 level8/9 短板，不能直接和只测 `[4,5,6,7]` 的 P7 高难口径分数横比。

#### C0-P7.3-goal-aware-local-waypoint

P7.3 从 P7.1-11650 作为起点，不先改 reward、不改模型维度，只修正 rule-level waypoint。触发原因来自 replay：迷宫里左右都能走时，策略会选择更远的一侧；上下台阶时也会斜着上/下，而不是沿赛道方向直走。

代码层判断：这更像高层 command/waypoint 的偏置，而不是单纯 standard 预训练 gait bias。旧逻辑在前方 blocked 时只看 `height_scanner` 左右障碍比例：如果左右近似对称，也会落到固定侧向 waypoint。这样在 maze 分叉会忽略最终 goal 的左右方向，在台阶/坡面这种对称高度变化上也容易被误触发成横向绕行。

P7.3 改动如下：

- `TrackNavConfig.lr` 从 P7.2 的 `5e-5` 恢复为 P7.1 的 `1e-4`，撤销低学习率实验。
- `_height_scanner_navigation_hint` 不再直接返回固定 `side_sign`，而是返回 `front_blocked/right_blocked/left_blocked` 三个诊断量。
- 新增 `_choose_local_waypoint_y`：左右障碍差异明显时仍走更空的一侧；左右近似对称时，local waypoint 的横向偏移改用最终 goal 在机体系下的 `final_y_body`，并 clamp 到 `[-1.2, 1.2]`。

预期效果：对称台阶/坡面不再被强行侧向绕行，应该更直；maze 分叉处如果两边都通，优先按最终 goal 方向选边，减少“明明左边近却去右边远路”的行为。该版本应先做 rule-only eval：直接用 P7.1-11650 checkpoint 跑 `[4,5,6,7]`，不要先训练。如果 rule-only 不掉分或 replay 行为改善，再短训 10-step 粒度 checkpoint；如果直接掉分，回退到 P7.1-11650。
